# TermHub Agent Platform — 비전 및 계획

## 핵심 아이디어

**터미널 세션 = 에이전트. 프롬프트 템플릿으로 행동을 유발한다.**

하나의 Claude Code 세션이 이미 충분히 똑똑하다. 계획, 구현, 리뷰를 하나의 세션에서 맥락을 유지하며 수행할 수 있다. 별도의 에이전트 레이어, 오케스트레이션 엔진은 불필요하다.

## 왜 오케스트레이션이 필요 없는가

오케스트레이션이 해결하려던 문제들:
- "에이전트마다 다른 역할" → **프롬프트로 해결** (같은 세션에서 "리뷰해줘" 한마디면 됨)
- "결과를 다음 단계에 전달" → **같은 세션이니 이미 기억함** (맥락 유지)
- "순차/병렬 실행" → **긴 프롬프트 하나면 Claude가 알아서 순서대로 함**

멀티 에이전트가 필요한 경우는 "여러 프로젝트를 동시에 관리"하는 것뿐이고, 그건 탭 전환으로 충분하다.

## 구조

```
Layer 0: 세션 관리 (이미 있음)
  └─ 웹 터미널, 다중 세션, tmux, 프로젝트 런처
  └─ Claude/Codex 상태 감지, 알림
  └─ 브라우저, SSH, Remote Control로 접속

Layer 1: 프롬프트 템플릿 (추가할 것)
  └─ 자주 쓰는 프롬프트를 저장하고 원클릭 전송
  └─ 글로벌 템플릿 / 프로젝트별 템플릿
  └─ 웹 UI에서 CRUD + 세션에 전송
  └─ 체인(여러 템플릿 순차 전송)은 나중에 필요하면

Layer 2: 도구 관리 (나중에)
  └─ MCP 서버 등 도구를 세션별로 설정
  └─ 웹 UI에서 도구 연결/해제
```

## 프롬프트 템플릿 상세

### 템플릿 포맷
```markdown
---
name: review
description: 코드 리뷰 수행
---
변경된 코드를 리뷰해줘. 다음 기준으로:
- 보안 취약점
- 성능 이슈
- 코드 컨벤션 준수
P0/P1/P2로 분류해서 보고해줘.
```

### 저장소
- `~/.termhub/templates/*.md` — 글로벌 템플릿
- 프로젝트별 템플릿은 나중에 (필요 시)

### API
- `GET /api/templates` — 목록
- `POST /api/templates` — 생성
- `PUT /api/templates/:name` — 수정
- `DELETE /api/templates/:name` — 삭제
- `POST /api/sessions/:id/template/:name` — 세션에 템플릿 전송

### UI
- 액션 바에 템플릿 버튼 추가
- 템플릿 관리 화면 (CRUD)
- 세션에서 템플릿 선택 → 원클릭 전송

## 설계 원칙

1. **세션이 곧 에이전트**: 추상화 없이 터미널 세션 자체가 에이전트.
2. **프롬프트가 행동**: "설정"이 아니라 "전송"으로 행동을 유발. 언제든 다른 프롬프트 가능.
3. **맥락 보존**: 하나의 세션에서 여러 작업 수행. 맥락이 끊기지 않는다.
4. **점진적 복잡도**: 템플릿 하나 → 필요하면 체인 → 필요하면 자동화.
